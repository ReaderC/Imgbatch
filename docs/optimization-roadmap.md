# 待优化方案

这份文档用于记录后续两项高价值性能优化的设计思路：

- 图片列表虚拟滚动
- 页面局部更新

当前版本已经完成一轮低风险优化，包括：

- 队列缩略图缓存
- 导入后先显示列表信息，再异步补缩略图
- 处理进度按帧合并
- 队列错误信息精简展示
- 处理完成后清理遗留进度状态

在此基础上，后续如果还要继续提升“图片很多时的流畅度”和“点击切页时的响应速度”，优先考虑下面两项。

## 1. 图片列表虚拟滚动

### 目标

当右侧图片列表有很多项时，只渲染当前视口附近的卡片，而不是一次性渲染全部图片项。

### 当前问题

- 图片一多，右侧列表会生成大量 DOM。
- 当前架构下，点击切工具、修改配置、处理进度更新时，列表也会跟着整块参与重排和绘制。
- 即使部分卡片离开视口，仍然会增加 HTML 拼接、布局和绘制成本。

### 核心思路

- 队列列表容器保留完整滚动高度。
- 只渲染当前可见区和上下缓冲区内的项目。
- 根据滚动位置动态计算：
  - `startIndex`
  - `endIndex`
  - 上方占位高度
  - 下方占位高度

### 推荐落地方式

1. 给图片列表增加统一行高模型。
2. 先实现定高虚拟列表。
3. 视口缓冲建议从上下各 `6 ~ 10` 项开始。
4. 如果后续出现单项高度差异，再补可变高度支持。

### 第一阶段范围

- 仅对右侧普通工具页的图片列表启用虚拟滚动
- 不碰手动裁剪页
- 不碰结果页
- 不改变现有排序、删除、预览按钮行为

### 风险点

- 拖拽排序和虚拟滚动同时存在时，需要额外处理目标索引映射。
- 滚动定位、删除当前项、批量处理完成后的自动回显，都要确认不会跳位。

### 预期收益

- 图片很多时，滚动和点击切页的流畅度会明显提升。
- DOM 数量下降后，全屏模式下的布局和绘制压力会进一步降低。

## 2. 页面局部更新

### 目标

避免每次状态变化都整页重绘 `#app`，改成只更新受影响的区域。

### 当前问题

- 现在的主渲染仍然是整页 `innerHTML` 重建。
- 即使只更新处理进度、切换顶部按钮文案，右侧图片列表和中间页面也会一起重绘。
- 这会导致：
  - 点击导航切换页面慢半拍
  - 调整配置按钮时界面响应偏重
  - 图片越多越明显

### 推荐拆分区域

建议把应用拆成固定壳和多个独立更新区域：

- 左侧导航区
- 顶部栏
- 中间工作区
- 右侧图片列表
- 覆盖层（弹窗、通知、预览）

### 推荐落地顺序

1. 先固定外层壳结构，只初始化一次。
2. 第一步先拆：
   - 顶部栏局部更新
   - 右侧列表局部更新
3. 第二步再拆中间工作区。
4. 最后再拆通知、弹窗、预览等覆盖层。

### 第一阶段建议

优先做这两块局部更新：

- `topbar`
- `queue`

原因：

- 风险低于直接重构整个页面
- 对“处理中点击按钮卡顿”的改善最直接

### 风险点

- 需要引入更明确的“状态变更影响范围”判断。
- 现有的 tooltip、snapshot、post-render 同步逻辑要重新挂接到局部区域上。

### 预期收益

- 点击左侧导航、切换配置页、开关顶部按钮时，响应会更轻。
- 即使列表较长，也不会因为一次小状态变化而强制整页重建。

## 建议优先级

如果后续要继续优化，推荐顺序：

1. 图片列表虚拟滚动
2. 顶部栏 + 右侧列表局部更新
3. 中间工作区局部更新
4. 覆盖层和后处理逻辑继续细化

## 备注

当前版本已经比最初明显顺畅，后续优化重点不再是继续零碎修补，而是进入结构优化阶段。
